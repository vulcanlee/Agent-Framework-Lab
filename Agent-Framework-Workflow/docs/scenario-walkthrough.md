# 情境走讀

## 為什麼選「週末旅遊行程規劃」

這是一個非常生活化的需求，幾乎每個人都能立即理解。

它也同時具備兩種特性：

- 有些部分很適合交給模型自由發揮
- 有些部分其實是很典型的固定流程

所以它很適合拿來比較 Agent 與 Workflow。

## 使用者輸入

輸入通常像這樣：

```text
幫我規劃兩天一夜台中週末旅行，兩個大人同行，每人預算 3500 元，希望行程輕鬆、有咖啡店，也想吃在地美食。
```

## Agent 路線

Agent 路線的思考方式是：

- 把需求交給一個旅遊規劃助理
- 若需要更具體資訊，再決定是否呼叫工具
- 最後一次產生完整回答

在這條路線上，工具只是輔助：

- 交通建議
- 打包建議
- 預算提醒

真正負責統整的人是 Agent 本身。

## Workflow 路線

Workflow 路線把同一件事拆成可理解的步驟。

### Step 1: 需求正規化

把自然語言需求轉成穩定欄位：

- 目的地
- 天數
- 同行人數
- 每人預算
- 旅遊風格
- 限制條件

### Step 2: 預算檢查

根據預算決定：

- 行程是否要偏平價
- 要不要加上警示
- 是否應避免高成本安排

### Step 3: 景點規劃

模型只負責產生景點安排草案。

### Step 4: 餐食規劃

模型根據目前的景點草案，再補上餐食建議。

### Step 5: 最後組裝

最後一步不是再丟給模型，而是由程式把各步驟結果拼成最終輸出。

這代表開發者保留了最終格式控制權。

## 為什麼這樣設計

這個拆法有兩個教學價值：

1. 初學者很容易看出每一步的責任。
2. 很容易延伸成技術文章，因為每一步都能獨立解釋。

如果之後你要擴充，方向也很清楚：

- 在預算檢查後加入住宿建議
- 在景點規劃後加入天氣檢查
- 在最後組裝前加入人工審核

這就是 Workflow 相對於單一 Agent 的最大優勢：它不是只有「會不會回答」，而是讓你明確定義「怎麼回答」。
