# Agent 與 Workflow 的差異

這個範例故意用同一個「週末旅遊行程規劃」情境，同時示範 Agent 與 Workflow。

## Agent 版本在做什麼

Agent 版本把大部分決策交給模型：

- 理解使用者需求
- 決定回覆結構
- 視需要呼叫交通、打包、預算工具
- 自己把內容整理成可閱讀的行程

這種做法的優點：

- 寫起來快
- 使用者體驗自然
- 適合需求模糊、變化大的互動任務

代價是：

- 每次回覆的形狀可能不完全一致
- 如果提示詞不夠清楚，容易過度發散
- 可治理性與可預測性較弱

## Workflow 版本在做什麼

Workflow 版本把流程拆成明確步驟：

1. 需求正規化
2. 預算檢查
3. 景點規劃
4. 餐食規劃
5. 最後組裝輸出

這種做法的優點：

- 責任邊界清楚
- 每一步都能單獨替換或測試
- 較容易插入規則、審核與觀測

代價是：

- 設計成本較高
- 一開始會比單一 Agent 寫得更長
- 若拆分過細，反而會讓程式變得笨重

## 什麼時候用哪一個

偏向用 Agent 的情況：

- 問題描述模糊
- 需要比較自然的互動
- 工具使用條件很難事先寫死

偏向用 Workflow 的情況：

- 流程本來就固定
- 每一步都需要可觀測與可治理
- 你要控制每個節點何時執行、輸入是什麼、輸出長什麼樣子

## 這個專案想表達的實務觀點

不要把「Agent」當成比較進階，所以什麼都拿來做 Agent。

比較務實的判準是：

- 如果一般函式與固定流程就能解決，不要硬上 Agent。
- 如果需求有模糊判斷與工具選擇，再考慮讓 Agent 參與。

很多真實系統最後會採混合做法：

- 用 Agent 理解需求與決定策略
- 用 Workflow 實際執行核心流程

這也是 Microsoft Agent Framework 很適合教學的原因，因為它不是只給你一種抽象，而是讓你能在 Agent 與 Workflow 之間做清楚分工。
